# MINIGAME YOLO Boundaries

本文件记录 MINIGAME 项目中允许 Hermes Agent 自主推进时必须遵守的边界。

## 默认工作目录

- 默认仅在本项目目录内工作：`D:\All projects\MINIGAME`
- 所有代码、测试、构建、预览、提交操作默认限制在该项目目录内。
- 测试文件、临时文件、脚本输出、缓存和构建产物不得溢出到非工作区。
- 运行命令时必须显式使用本项目作为工作目录，避免测试或脚本在用户电脑其他位置创建文件。
- 如必须生成临时文件，默认放入项目内的 `.tmp/`、`tmp/`、`test-output/` 或同类项目内目录，并在提交前确认是否应删除或加入 `.gitignore`。

## 允许自主执行

- 修改本项目源码、样式、测试、构建脚本和文档。
- 运行本项目测试、构建、预览服务。
- 创建小步、可回滚的 Git commit。
- 做浏览器预览和交互验收。

## 禁止访问 / 修改 / 删除

以下是硬边界，除非用户在当前会话中明确临时授权，否则禁止执行：

1. **C 盘 Windows 系统**
   - 不修改 `C:\Windows` 或其他 Windows 系统目录。
   - 不改系统服务、注册表、驱动、系统 PATH、系统级安全/代理/网络设置。

2. **D 盘软件**
   - 不修改或删除 D 盘上的软件目录、安装目录、工具目录。
   - 对 D 盘的操作默认只限当前项目：`D:\All projects\MINIGAME`。

3. **E 盘**
   - 禁止访问 E 盘。
   - 禁止读取、搜索、修改、删除 E 盘任何内容。

4. **非工作区文件溢出**
   - 禁止把测试文件、临时文件、下载物、脚本输出、构建中间产物写到项目目录之外。
   - 禁止使用会在用户主目录、桌面、系统临时目录、其他磁盘目录散落文件的测试/脚本流程。
   - 如果第三方工具默认写到项目外，必须先改配置或环境变量让输出落在项目内；做不到则停止并询问用户。

## Git 边界

- 可以 `git status` / `git diff` / `git add` / `git commit`。
- 不得主动 `git push`，除非用户明确要求。
- 不得执行破坏性 Git 操作，例如：
  - `git reset --hard`
  - `git clean -fd`
  - `git rebase`
  - 改写历史

## 停止并询问的情况

遇到以下情况必须停下来问用户：

- 需要访问项目目录之外的磁盘路径。
- 测试、构建或脚本可能向项目目录之外写文件。
- 需要修改系统或软件安装配置。
- 需要新增大型依赖或重构构建系统。
- 需要真实广告、发布、账号、密钥、远程推送。
- 自动化测试连续修复 3 次仍失败。

## 每轮 YOLO 验收标准

- 代码变更后运行相关测试。
- 涉及 H5/浏览器时必须打开预览页实际检查。
- UI 是硬质量门槛：界面必须美观、有游戏感，符合暗色异常控制台 / HUD / CCTV / 终端日志氛围；不能像表单、后台管理页或调查问卷。
- 涉及 UI 的改动必须做浏览器视觉验收，检查布局层级、按钮质感、状态反馈、遮罩弹层、移动端可读性和整体审美。
- 涉及微信包时运行 `node build.js wechat` 和 `node --check wechat-minigame/game.js`。
- 提交前检查 `git status --short` 和 `git diff --stat`。
